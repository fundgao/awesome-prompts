# Next.js 16 + next-intl 多语言切换（zh-CN / zh-TW / en）AI 编程文档

## 1. 背景与目标

### 1.1 目标

在 Next.js 16（App Router）项目中，基于 `next-intl` 实现三语言切换：

* 简体中文：`zh-CN`
* 繁体中文：`zh-TW`
* 英文：`en`

并在 **导航栏右侧** 放置一个“语言切换按钮（icon）”，点击后可切换语言并保持当前页面路径。

### 1.2 默认语言策略

默认语言取 **用户浏览器语言**（Accept-Language），并满足：

* 首次访问（无偏好记录）时：按浏览器语言自动进入最匹配语言
* 用户手动切换语言后：记住偏好（cookie），后续访问优先 cookie，而不是浏览器语言
* 不支持的语言：回退到 `en`

### 1.3 非目标（明确不做）

* 不做“自动翻译”
* 不做动态从 CMS 拉取文案（仅本地 messages）
* 不做复杂的地区细分（如 en-GB、zh-HK），仅上述三种 locale

---

## 2. 关键约束（必须遵守）

* 使用 `next-intl`（App Router 方案）
* 语言切换必须保持当前页面的 pathname 与 query（如果有）
* Locale 必须是：`en` / `zh-CN` / `zh-TW`
* 默认语言逻辑：cookie 优先，其次 Accept-Language，最后回退 `en`
* 语言切换按钮：放在导航栏 **右侧**，使用 icon（可用现有 icon 库，如 `lucide-react`；若项目已有 icon 组件则复用）
* 兼容 SSR / RSC：不要在 Server Component 里直接使用浏览器 API（`navigator.language` 等）

---

## 3. 路由与国际化方案设计

### 3.1 路由策略（推荐：Locale 前缀路由）

使用 locale 前缀，示例：

* `/en/...`
* `/zh-CN/...`
* `/zh-TW/...`

理由：与 next-intl App Router 推荐模式一致，页面切换时只替换第一段 locale，容易保持路径。

> 若你项目当前没有 locale 前缀路由：本方案会引入 `app/[locale]/...` 结构，并通过 middleware 在访问 `/` 或无 locale 前缀时重定向到检测到的 locale。

### 3.2 Locale 检测与重定向

使用 Next.js middleware 做检测：

优先级：

1. `NEXT_LOCALE` cookie（next-intl 默认 cookie 名可用，也可以你自定义，但建议沿用 next-intl 的惯例）
2. `Accept-Language` header 匹配（例如浏览器是 `zh-TW` → `zh-TW`；`zh` → 默认映射 `zh-CN`）
3. fallback：`en`

行为：

* 当请求路径不包含 locale（如 `/`, `/about`）时 → 302 redirect 到 `/<detected-locale>/<原路径>`
* 当路径已包含 locale（如 `/en/about`）时 → 不重定向

### 3.3 消息文件组织

建议放在：

```
/messages
  en.json
  zh-CN.json
  zh-TW.json
```

key 结构尽量稳定，例如：

```json
{
  "nav": {
    "home": "Home",
    "docs": "Docs"
  },
  "language": {
    "switch": "Switch language",
    "en": "English",
    "zhCN": "简体中文",
    "zhTW": "繁體中文"
  }
}
```

## 4. UI 设计：导航栏右侧语言切换 icon

### 4.1 交互要求

* 导航栏右侧有一个 icon 按钮（例如 globe 图标）
* 点击后展示一个小菜单/下拉（3 个选项：简体/繁体/English）
* 选择其中一个后：

  * 切换到对应 locale
  * 保持当前 pathname（除了 locale 段）
  * 设置 cookie 记住语言偏好

### 4.2 可访问性（必须）

* icon button 必须有 `aria-label`（例如：`Switch language`）
* 菜单项可键盘操作（如果你用现成 UI 组件库优先复用）

### 4.3 组件建议

* `LanguageSwitcher` 做成 Client Component（因为要响应点击、路由跳转）
* Navbar 可以是 Server 或 Client，但 LanguageSwitcher 必须是 Client

---

## 5. 技术实现细节（next-intl App Router）

### 5.1 i18n 配置（示例约定）

* locales：`['en', 'zh-CN', 'zh-TW']`
* defaultLocale（逻辑上的 fallback）：`en`

建议新增文件：

* `src/i18n/routing.ts`：定义 locales、localePrefix 等
* `src/i18n/request.ts`：next-intl 的 `getRequestConfig`（加载 messages）

### 5.2 Middleware（核心）

middleware 做两件事：

1. 若路径无 locale → 检测 locale 并 redirect
2. 与 next-intl middleware 集成（如果使用 next-intl 提供的 createMiddleware 更省事）

匹配规则（matcher）要避开：

* `/_next/*`
* 静态资源（如 `.png`, `.ico`, `.svg`）
* API routes（如 `/api/*`）

### 5.3 页面与 Layout 结构

新增：

```
app/[locale]/layout.tsx
```

* 从 params 里拿 locale
* 设置 `<html lang="...">`
* 在 layout 内加载 messages provider（next-intl）

### 5.4 切换语言保持路径

在 LanguageSwitcher 中：

* 获取当前 pathname（Next `usePathname()`）
* 将第一段 locale 替换为目标 locale
* push 到新路径（`useRouter().push(newPath)`）
* 同时写 cookie（`document.cookie = "NEXT_LOCALE=...; path=/; max-age=..."`）
  或者用一个 `/api/locale`/server action 来写 cookie（更干净，但会多一个请求；本需求允许前端写 cookie 即可）

## 6. 验收标准（Acceptance Criteria）

### 6.1 功能

* [ ] 首次访问 `/`：会 302 到基于浏览器语言的 `/<locale>`

  * 浏览器 `zh-TW` → `/zh-TW`
  * 浏览器 `zh` 或 `zh-CN` → `/zh-CN`
  * 浏览器 `en` 或其他未知 → `/en`
* [ ] 访问任意无 locale 前缀路径（如 `/docs`）会自动跳到 `/<detected-locale>/docs`
* [ ] 在任意页面 `/<locale>/...`，导航栏右侧有语言 icon
* [ ] 点击 icon 可选择三种语言
* [ ] 切换语言后：路径除 locale 段以外保持不变（query 也保持）
* [ ] 切换语言后：刷新页面语言保持（cookie 生效）
* [ ] 不支持的 Accept-Language 回退到 `en`

### 6.2 体验与质量

* [ ] icon button 有 aria-label
* [ ] 没有明显的 hydration 报错
* [ ] middleware matcher 不影响静态资源与 `_next` 资源加载

## 9. 风险点与注意事项

* Next App Router 下，`usePathname/useRouter` 只能在 Client Component 使用
* middleware 的 matcher 写错会导致静态资源 404 或无限重定向
* Accept-Language 匹配要处理：

  * `zh` → 映射到 `zh-CN`
  * `zh-Hant`（部分浏览器）→ 优先 `zh-TW`（可做简单映射）
* 切换语言时 URL 结构要统一，否则会出现“切换后跳错路由”的问题
